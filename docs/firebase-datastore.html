Séance 1 :

Intro

Nous discuterons des principes de base du Firestore et des parallèles à faire entre un BD relationnelle et Firestore qui est orienté documents/collections

Rappels : Service/Repo, entités/DTO

La couche firestore correspond à votre repository. Reste maintenant à implanter les validations, contrôles d'accès, tri, règles applicatives etc.

Pour ces différentes raisons, on vous recommande de vous faire un service.dart où on regroupera les différentes méthodes de service. On vous suggère de faire ainsi:

  • Le seul endroit où on appelle les méthodes de firestore, c'est dans service.dart
  • Quand l'interface graphique veut faire quelque chose, elle appelle une fonction de service.dart

Revenir sur différence entre modèle et classes de transfert (DTO). On fait du modèle ici.

Comment relier l'utilisateur à ses tâches

Nous avons des utilisateurs qui ont chacun des tâches dans leur compte. Comme on utilise des comptes Google, nous n'avons pas forcément besoin d'une collection d'utilisateurs.

  • Une collections d'utilisateurs, chaque utilisateur contient une collections de tâches
  • Créer un compte dans le datastore qui utilise un identifiant de l'utilisateur
  • uniquement inclure l'identifiant dans les objets stockés dans le datastore
